# Web 可访问性标准 ## 核心原则 (POUR) - **可感知**：信息必须以用户可感知的方式呈现给他们 - **可操作**：用户界面组件必须可供所有用户操作 - **可理解**：信息和操作必须可理解 - **健壮**：内容必须足够健壮才能与当前和未来的技术协同工作 ## 合规级别 - **目标合规性**：WCAG 2.1 AA（最低） - **追求**：可行的情况下为 WCAG 2.1 AAA - **法律要求**：ADA、第 508 条、EAA（欧盟）、AODA（加拿大）（如适用） - **定期测试**：自动和手动可访问性测试 ## 语义结构 ### HTML 语义 - 使用适当的 HTML 元素来实现其预期目的 - 按逻辑顺序实现适当的标题结构（h1-h6） - 使用地标元素（`<main> `, `<nav> `, `<header> `, `<footer> `, `<aside> `) - 实现适当的列表结构 (`<ul> `, `<ol> `, `<dl> `) - 使用 ` <button>` 进行交互式控制，而不是 div 或 span - 使用 ` <a>` 导航到新页面或资源 - 使用 `</a> <table>` 表示具有适当标题的表格数据 - 除非必要，否则避免使用 ARIA 覆盖本机语义 ### 文档结构 - 每页包含一个 `</table> <h1>` - 实现逻辑标题层次结构 - 使用描述性页面标题 - 为键盘用户提供跳过链接 - 使用 ` <fieldset>` 和 ` <legend>` 将相关表单元素分组 - 使用 ` <label>` 元素作为表单控件 - 实现适当的文档语言 (` <html lang="en">`) - 使用 ` <section>`、` <article>` 和其他语义容器 ## 键盘可访问性 - 确保所有交互式元素都可以通过键盘访问 - 实现逻辑 Tab 顺序（明智地使用 tabindex） - 提供可见的焦点指示器（增强默认样式） - 确保避免键盘陷阱 - 为复杂界面实现键盘快捷键 - 仅使用键盘测试所有功能 - 确保自定义组件支持键盘交互模式 - 支持标准键盘交互（空格键、回车键、Esc 键、箭头键） ## 图像和媒体 ### 图像 - 为所有图像提供替代文本（`alt` 属性） - 对装饰性图像使用空 alt（`alt=&quot;&quot;`） - 为信息丰富的图片提供描述性替代文本 - 对复杂图像（图表、图形）使用适当的文本替代 - 确保文本未嵌入图像中 -<figure><figcaption>对带有标题的图片使用 ` ` 和 ` ` - 为重要图片保持足够的对比度 ### 媒体 - 为视频提供标题 - 为音频内容提供文字记录 - 需要时为视频实现音频描述 - 确保媒体播放器可通过键盘访问 - 提供暂停、停止和调整音量的控件 - 避免自动播放有声媒体 - 确保内容每秒闪烁不超过 3 次 ## 颜色和对比度 - 确保足够的颜色对比度： - 普通文本 (AA) 为 4.5:1 - 大文本 (AA) 为 3:1 - 普通文本 (AAA) 为 7:1 - 大文本 (AAA) 为 4.5:1 - 不要仅依靠颜色来传达信息 - 提供其他指示（图标、图案、文本） - 以灰度测试界面 - 支持高对比度模式 - 确保焦点指示器具有足够的对比度 - 使用色盲模拟器进行测试 - 使用可访问的颜色组合 ##表单和交互元素 ### 表单 - 将标签与表单控件关联（使用 `for` 属性） - 使用 `</figcaption></figure></article></section></html></label></legend> <fieldset>` 和 ` <legend>` 将相关表单元素分组 - 提供清晰的说明和错误消息 - 指示必填字段（不仅仅是用颜色） - 验证输入并提供可访问的错误消息 - 验证错误后保持表单状态 - 支持自动完成属性 - 使用适当的输入类型（电子邮件、电话、日期等） ### 交互元素 - 确保所有控件都有可访问的名称 - 提供足够大的触摸目标（至少 44×44px） - 在需要时实现 ARIA 角色、状态和属性 - 确保自定义组件遵循 WAI-ARIA 创作实践 - 为所有交互提供反馈 - 确保状态变化通知屏幕阅读器 - 使用辅助技术测试复杂的小部件 ## 焦点管理 - 保持可见的焦点指示器 - 切勿在没有替代方案的情况下移除焦点轮廓 - 在内容动态变化时管理焦点 - 操作后将焦点返回到适当的元素 - 在模态框和对话框中捕获焦点 - 为单页应用程序实现程序化焦点 - 确保焦点顺序与视觉顺序匹配 - 适当使用 `tabindex`（避免使用正值）##文本和排版 - 使用相对文本单位 (rem, em) - 确保文本大小可调整至 200% 而不会丢失内容 - 段落文本的行高至少为 1.5 - 行长保持在 50-75 个字符之间 - 段落之间留出足够的间距 - 尽可能使用清晰、简单的语言 - 避免使用两端对齐的文本 - 确保文本和背景之间有适当的对比度 - 支持文本间距调整 ## ARIA 实施 - 尽可能在 ARIA 之前使用原生 HTML 元素 - 为页面结构实现 ARIA 标志 - 需要时使用 ARIA 角色定义元素类型 - 对动态内容应用 ARIA 状态和属性 - 内容更改时更新 ARIA 状态 - 使用屏幕阅读器测试 ARIA 实施 - 遵循 WAI-ARIA 创作实践 - 避免冲突或冗余的 ARIA ## 动态内容和 AJAX - 向屏幕阅读器宣布动态内容更改 - 适当使用 ARIA 实时区域 - 内容更新时管理焦点 - 为异步内容提供加载指示器 - 确保自定义滚动区域可访问 - 使单页应用程序可导航 - 在后退/前进导航期间保持页面状态 - 确保浏览器历史记录已正确更新 ## 测试和验证 ### 自动化测试 - 在开发工作流程中集成可访问性 linting - 使用 axe-core 或类似工具进行自动化测试 - 在 CI/CD 管道中包含可访问性测试 - 生成可访问性报告 - 跟踪可访问性问题随时间的变化 - 使用多种自动化工具进行测试 - 了解自动化测试的局限性 ### 手动测试 - 仅进行键盘测试 - 使用屏幕阅读器测试（NVDA、JAWS、VoiceOver） - 使用屏幕放大镜测试 - 使用语音识别软件测试 - 手动验证颜色对比度 - 使用浏览器放大至 200% 进行测试 - 使用不同的显示尺寸进行测试 ### 用户测试 - 将残疾用户纳入测试 - 与辅助技术用户一起测试 - 收集有关可访问性问题的反馈 - 根据影响确定修复的优先级 - 记录已知问题的解决方法 - 定期进行可访问性审查 - 实施持续改进 ## 文档和培训 - 记录可访问性功能 - 提供可访问性声明 - 对开发团队进行可访问性培训 - 在设计评审中包括可访问性 -
